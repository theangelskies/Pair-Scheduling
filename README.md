# Pair-Scheduling

A MigraCode fullstack project built with:

- **Frontend:** Vite 8 + React 19 + TanStack Router (file-based routing) + CSS Modules + TypeScript 6
- **Backend:** Node.js + Express 5 + TypeScript 6
- **Database:** PostgreSQL 17 (Docker)
- **Testing:** Vitest 4 + Testing Library
- **Linting:** oxlint
- **Formatting:** oxfmt
- **CI:** GitHub Actions


### Adding a new table

Say you want to add a `posts` table. Here are the exact steps:

**1. Create a new migration file:**

```bash
cd backend
npm run migrate:create -- add-posts-table
```

This creates a file like `backend/migrations/1749823642869_add-posts-table.js` (the number is a timestamp). Open it — it looks like this:

```js
export const shorthands = undefined

export const up = (pgm) => {
  // TODO: write your changes here
}

export const down = (pgm) => {
  // TODO: write how to undo the changes here
}
```

**2. Fill in `up` and `down`:**

```js
export const shorthands = undefined

export const up = (pgm) => {
  pgm.createTable('posts', {
    id: { type: 'serial', primaryKey: true },
    title: { type: 'text', notNull: true },
    body: { type: 'text', notNull: true },
    created_at: { type: 'timestamp', default: pgm.func('NOW()'), notNull: true },
  })
}

export const down = (pgm) => {
  pgm.dropTable('posts')
}
```

- `up` — what to do when migrating forward (create table, add column, etc.)
- `down` — how to undo it (used by `migrate:down` to roll back)

**3. Run the migration:**

```bash
npm run migrate
```

node-pg-migrate tracks which migrations have already run (in a `pgmigrations` table), so it only applies new ones.

### Resetting the database

To wipe everything and start fresh:

```bash
docker compose down -v         # stop PostgreSQL and delete the data volume
docker compose up -d           # restart with a clean database
cd backend && npm run migrate  # recreate all tables and seed data
```

### Other database commands

```bash
npm run migrate:down           # roll back the last migration (run from backend/)
docker exec -it Pair-Scheduling-db psql -U postgres -d appdb   # open a psql shell
docker compose down            # stop PostgreSQL (keeps data)
```

## TanStack Router

Routes live in `frontend/src/routes/`. The Vite plugin **auto-generates** `routeTree.gen.ts` whenever you add or rename route files — never edit that file manually.

**Adding a new page:**

1. Create `frontend/src/routes/my-page.tsx`
2. Export a `Route` using `createFileRoute('/my-page')({...})`
3. Add a `<Link to="/my-page">` in `__root.tsx`

The file name determines the URL path automatically.

## Styling (CSS Modules)

Each component has its own `.module.css` file next to it. Styles are **scoped** — a `.card` class in `UserCard.module.css` will never clash with a `.card` class in another component.

```
frontend/src/
├── components/
│   ├── UserCard.tsx
│   └── UserCard.module.css    ← scoped styles for UserCard only
```

Import and use them in your component:

```tsx
import styles from './UserCard.module.css'

export function UserCard({ name }) {
  return <div className={styles.card}>{name}</div>
}
```

Vite hashes the class names at build time so they can never collide. Global styles (resets, fonts) live in `frontend/src/index.css`.

## Architecture

The project enforces a strict separation of concerns — keeping each layer focused on one job:

**Frontend**

- `components/` — small, reusable UI pieces (props in, HTML out)
- `routes/` — one file per URL; the file name IS the URL path

**Backend**

- `routes/` — HTTP only: read the request, call a service, send the response
- `services/` — all business logic; no knowledge of HTTP or Express
- `db/` — database connection pool and migrations

Keeping routes thin and services pure means you can test services without spinning up a server, and swap the framework without rewriting logic.

## Deploying

The easiest path for a bootcamp project:

| What              | Where                                                                                          | Notes                                             |
| ----------------- | ---------------------------------------------------------------------------------------------- | ------------------------------------------------- |
| Frontend (static) | [Vercel](https://vercel.com) or [Netlify](https://netlify.com)                                 | Free tier, deploy straight from GitHub            |
| Backend (Node.js) | [Railway](https://railway.app) or [Render](https://render.com)                                 | Free tier available                               |
| PostgreSQL        | [Railway](https://railway.app), [Neon](https://neon.tech), or [Supabase](https://supabase.com) | Managed Postgres — no Docker needed in production |

**Steps:**

1. **Deploy the backend** to Railway or Render. Set these environment variables on the platform:

   ```
   DATABASE_URL=<your production database URL>
   PORT=3000
   ```

2. **Run migrations** on the production database (once after first deploy, then after each schema change):
   Most platforms let you run a one-off command in the shell — run `npm run migrate` from the `backend/` directory.

3. **Deploy the frontend** to Vercel or Netlify. Point it at your repository's `frontend/` folder.
   Set the build command to `npm run build` and the output directory to `dist`.

4. **Connect frontend to backend**: In development, the Vite proxy forwards `/api` requests to `localhost:3000` automatically. In production you have two options:
   - **Same domain** (recommended): configure your hosting platform to route `/api/*` to your backend — then your frontend code works unchanged.
   - **Different domains**: add `VITE_API_URL=https://your-backend.com` as an env var on Vercel/Netlify, then prefix all `fetch` calls with `import.meta.env.VITE_API_URL ?? ''`.
